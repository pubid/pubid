# V2 Architecture Guide

This document describes the PubID V2 architecture: design principles, parse pipeline, identifier class hierarchy, and how flavors are structured.

== Design Principles

1. **Single gem**: All flavors live in one gem under `lib/pubid/{flavor}/`
2. **Model-driven**: Identifier objects are Lutaml::Model serializable classes
3. **Self-describing types**: Each identifier class IS the type definition; no central registry required
4. **Renderer pattern**: Output formatting is separated into dedicated Renderer classes
5. **Single source of truth**: Series matching, type metadata, and rendering live in one place
6. **Open/Closed**: New flavors add files, not modify existing ones
7. **Round-trip fidelity**: Parse then render produces the original string
8. **Parslet PEG parsers**: Deterministic parsing with composable grammar rules
9. **Pre-parse normalization**: Malformed/legacy identifiers are fixed before parsing
10. **Autoload**: Lazy loading via Ruby autoload defined in parent namespace files

== Parse Pipeline

[source]
----
Input String
    |
    v
Pre-parser Normalization (data/{flavor}/update_codes.yaml)
    |
    v
Parser (Parslet PEG grammar -> parse tree)
    |
    v
Builder (parse tree -> attribute hash)
    |
    v
Identifier (Lutaml::Model object -> render(format:) -> to_s / to_urn / to_h)
----

=== Pre-parser Normalization

Applied *before* parsing. Maps malformed or legacy identifier strings to canonical form.

- Data files: `data/{flavor}/update_codes.yaml`
- Supports exact match and regex patterns
- Handles legacy formats, typos, and historical variations
- Currently available for: ISO, IEC, IEEE, NIST, CCSDS, PLATEAU

=== Parser Layer

Each flavor defines a Parslet PEG grammar in `parser.rb`:

- Rules are composable and deterministic
- `TYPED_STAGES` from identifier classes are flattened into parser rules
- Abbreviations are sorted by length (longest first) for correct PEG matching
- Parser returns a raw parse tree (hash)

=== Builder Layer

Builders transform parse trees into attribute hashes and instantiate identifier objects:

- Each flavor has a `builder.rb` that delegates to model classes
- NIST's builder is split into three: `Builder` (orchestrator), `Router` (series-to-class routing), `Caster` (type coercion) — single source of truth for series matching
- Builders accept parsed hash and construct identifiers with correct attribute assignment

=== Rendering Layer (Renderer Pattern)

All flavors use the Renderer pattern for human-readable output:

- Each flavor registers a dedicated `Renderer` class via `FormatRegistry`
- Per-flavor registries inherit from the global `Pubid::Identifier.format_registry` (shared `:mr_string` and `:urn` formats; overridden `:human`)
- Identifier `to_s(**opts)` delegates to `render(format: :human, **opts)`
- Adding a new output format (e.g., BibXML, JSON-LD) requires only a new Renderer class and registry entry — zero changes to identifier classes

[source,ruby]
----
# lib/pubid/iec/renderer.rb
module Pubid::Iec
  class Renderer < Pubid::Renderers::Base
    def render(context: nil, **opts)
      # IEC-specific composition
    end
  end
end

# lib/pubid/iec.rb — registry setup
Iec::Identifier.format_registry = FormatRegistry.new(parent: Identifier.format_registry)
Iec::Identifier.format_registry.register(:human, renderer: Iec::Renderer)
----

== Self-Describing Types

Each identifier type class IS the type definition. No central Scheme registry required.

[source,ruby]
----
# lib/pubid/iso/identifiers/international_standard.rb
class InternationalStandard < SingleIdentifier
  def self.type
    { key: :is, title: "International Standard", short: "IS" }
  end

  TYPED_STAGES = [
    TypedStage.new(abbr: "DIS", name: "Draft International Standard", ...),
    # ...
  ]
end
----

Each flavor module exposes model-driven lookup methods:

[source,ruby]
----
module Pubid::Iso
  def self.identifier_types
    @identifier_types ||= Identifiers.constants
      .map { |c| Identifiers.const_get(c) }
      .select { |c| c.is_a?(Class) && c.respond_to?(:type) }
      .select { |c| c.type.is_a?(Hash) }
  end

  def self.all_typed_stages
    @all_typed_stages ||= identifier_types.flat_map do |klass|
      klass.const_defined?(:TYPED_STAGES) ? klass.const_get(:TYPED_STAGES) : []
    end
  end

  def self.locate_type(code)
    identifier_types.find { |t| t.type[:key].to_s == code.to_s }
  end

  def self.locate_stage(abbr)
    all_typed_stages.find { |s| s.abbr.any? { |a| a.to_s.upcase == abbr.to_s.upcase } }
  end
end
----

Adding a new identifier type requires: (a) creating the class, (b) adding it to the `Identifiers` namespace. No other files modified.

== Flavor Module Structure

Each flavor under `lib/pubid/{flavor}/` follows this pattern:

[source]
----
lib/pubid/{flavor}/
|-- parser.rb            # Parslet PEG grammar
|-- transformer.rb       # AST -> attribute hash (some flavors)
|-- builder.rb           # Parse tree -> identifier object
|-- identifier.rb        # Top-level entry point (parse method)
|-- identifiers.rb       # Namespace module with autoload entries
|-- identifiers/         # Type-specific classes (each is self-describing)
|   |-- base.rb
|   |-- international_standard.rb   # Defines .type, TYPED_STAGES
|   |-- technical_report.rb
|   +-- ...
|-- components.rb        # Namespace module with autoload entries
|-- components/          # Flavor-specific component classes
|-- renderer.rb          # Dedicated Renderer for human-readable output
|-- urn_generator.rb     # URN output
+-- urn_parser.rb        # URN input (ISO, IEC)
----

== Identifier Class Hierarchy

[source]
----

|-- attributes: number, part, subpart, date, edition, languages,
|              publisher, copublishers, type, stage
|-- methods: to_h, to_json, root, exclude, new_edition_of?, render(format:, **opts)

Flavor-specific (e.g., Pubid::Iso::SingleIdentifier)
|-- to_s delegates to render(format: :human, **opts)
|-- uses dedicated Renderer for output composition

Pubid::SupplementIdentifier
|-- attributes: base_identifier, supplement_type
|-- chained: amendment -> corrigendum -> ...
----

== Autoload Structure

Ruby autoload is used throughout for lazy loading:

- Autoload entries are defined in the **immediate parent namespace's file**
- A new identifier class only needs to be added to its namespace's autoload list
- Autoload triggers on first constant reference, so removing `require_relative` calls has no runtime impact
- Cross-flavor `require_relative` is replaced with public API calls (where available)

== NIST Special Structure

The NIST flavor's builder is split into three classes for separation of concerns:

[source]
----
lib/pubid/nist/
|-- builder.rb    # 952 lines — orchestrator (was 2272)
|-- router.rb     # 143 lines — series-to-class routing (single source of truth)
|-- caster.rb     # 1337 lines — type coercion (extracted from cast())
+-- scheme.rb     #  97 lines — registry + delegations (was 199)
----

* **Router** owns series-to-class matching, supplement detection, and `VALID_SERIES`. Both Builder and Scheme delegate to it.
* **Caster** handles the massive type coercion logic that was previously inlined in Builder's `cast()` method.
* **Builder** is a slim orchestrator that calls Router for class resolution and Caster for type assignment.
* **Scheme** retains the identifier class registry (`identifiers`, `typed_stages`) and delegates routing to Router.

== Typed Stages

Typed stages combine a document lifecycle stage with a document type. Defined as `TYPED_STAGES` arrays on identifier classes.

[source,ruby]
----
# Example: IEC International Standard typed stages
TYPED_STAGES = [
  TypedStage.new(name: "Preliminary Work Item", stage_code: :pwi,
                 abbr: ["PWI"], harmonized_stages: ["00.00"]),
  TypedStage.new(name: "New Proposal", stage_code: :np,
                 abbr: ["NP"], harmonized_stages: ["10.00"]),
  TypedStage.new(name: "Working Draft", stage_code: :wd,
                 abbr: ["WD"], harmonized_stages: ["20.00"]),
  TypedStage.new(name: "Committee Draft", stage_code: :cd,
                 abbr: ["CD"], harmonized_stages: ["30.00"]),
  # ...
]
----

Harmonized stage codes follow ISO/IEC harmonized stage numbering:

|===
| Code Range | Stage

| 00.xx | Preliminary work item
| 10.xx | New proposal
| 20.xx | Working draft
| 30.xx | Committee draft
| 40.xx | Approval stage (CDV, etc.)
| 50.xx | Final draft (FDIS, etc.)
| 60.xx | Published
| 90.xx | Review
| 95.xx | Withdrawal
|===

== Testing

Tests are organized per-flavor under `spec/pubid/{flavor}/`:

- `spec/pubid/{flavor}/identifiers/` - per-type specs
- `spec/pubid/{flavor}/fixtures/` - bulk pass/fail fixtures (one ID per line)
- `spec/integration/` - cross-gem compatibility tests

Run with:

[source,bash]
----
bundle exec rspec spec/pubid/iec/              # All IEC tests
bundle exec rspec spec/pubid/iso/identifiers/  # All ISO type tests
bundle exec rake test:all                      # Full suite
----

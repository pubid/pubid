== Architecture

=== Three-Layer Design

```
Input: "ITU-T G.989-1 (06/2016)"
    ↓
┌──────────────────────────┐
│    Parser Layer          │  Parslet grammar
│  (Syntax → Parse Tree)   │  - Sector patterns (R/T/D)
└────────┬─────────────────┘  - Series recognition
         │ Parse tree         - Date formats
         ↓
┌──────────────────────────┐
│    Builder Layer         │  Object construction
│  (Tree → Components)     │  - Sector/Series/Code
└────────┬─────────────────┘  - Date with month
         │ Model objects      - Supplement handling
         ↓
┌──────────────────────────┐
│    Identifier Layer      │  Lutaml::Model classes
│  (Components → String)   │  - Recommendation
└────────┬─────────────────┘  - Supplement types
         │
         ↓
Output: "ITU-T G.989-1 (06/2016)"
```

=== Component Architecture

All ITU identifiers use these shared components:

[cols="1,3"]
|===
| Component | Purpose

| `Sector`
| ITU sector (R, T, D) with proper rendering

| `Series`
| Letter series (G, H, V, etc.) from series registry

| `Code`
| Number with optional parts and subseries

| `Date`
| Year and month in ITU format (MM/YYYY or YYYY)

| `Language`
| Language codes (E, F, R, etc.)
|===

=== Identifier Class Hierarchy

```
PubidNew::Identifier (ultimate parent)
    │
    ├─ PubidNew::Itu::Identifier (flavor base)
    │   │
    │   ├─ Recommendation (base documents)
    │   │   Examples: ITU-T G.989-1, ITU-R V.574-5
    │   │
    │   └─ Supplement (base for amendments)
    │       ├─ Supplement (Suppl.)
    │       ├─ Amendment (Amd)
    │       └─ Corrigendum (Cor.)
```

== Identifier Types

=== Recommendation (100% Complete)

Base ITU identifier type for standards.

**Patterns:**
```
ITU-T G.989-1 (06/2016)    # With parts and date
ITU-R V.574-5              # ITU-R with version
ITU-T H.264                # Simple pattern
ITU-T M.3016.1             # Subseries
```

**Components:**
- Sector: R, T, or D
- Series: G, H, V, M, etc.
- Code: Number with optional parts
- Date: Month/Year or Year only

=== Supplement Types

==== Two Supplement Patterns

**1. Supplement with Base (with number)**
```
ITU-T G.989 Amd 1 (2004)
ITU-T Z.100 (1999) Cor. 1 (10/2001)
```

**2. Supplement Series-Only (no number)**
```
ITU-T H Suppl. 1
ITU-R R Suppl. 14
```

==== Amendment (91.2% Complete)

Amendments to base recommendations.

**Patterns:**
```
ITU-T G.989 Amd 1 (2004)           # With base number
ITU-T G.780/Y.1351 Amd 1 (2004)    # Combined (limitation)
```

**Known Limitation:** Combined identifiers (G.780/Y.1351) fail due to parser limitation (3 tests).

==== Corrigendum (92.5% Complete)

Technical corrections to base recommendations.

**Patterns:**
```
ITU-T Z.100 (1999) Cor. 1 (10/2001)    # With base date
ITU-T H Cor. 2                          # Series-only
```

**Known Limitation:** Combined identifiers (3 tests).

==== Supplement (100% Complete)

Additional material to base recommendations.

**Patterns:**
```
ITU-T H Suppl. 1           # Series-only
ITU-T G.989 Suppl. 3       # With base number
```

== Usage Examples

[source,ruby]
----
require 'pubid_new'

# Parse ITU-T Recommendation
id = PubidNew::Itu.parse("ITU-T G.989-1 (06/2016)")

id.code.parts.first       # => "1"
id.date.year              # => "2016"

id.to_s                   # => "ITU-T G.989-1 (06/2016)"

== Parser Architecture

=== Supplement Patterns

The parser handles two distinct supplement patterns:

**1. `supplement_with_base`**: For recommendations with numbers
```ruby
# Pattern: BASE + SUPPLEMENT_TYPE + NUMBER + DATE
"ITU-T G.989 Amd 1 (2004)"
"ITU-T Z.100 (1999) Cor. 1 (10/2001)"
```

**2. `supplement_series_only`**: For series-only supplements
```ruby
# Pattern: SECTOR + SERIES + SUPPLEMENT_TYPE + NUMBER
"ITU-T H Suppl. 1"
"ITU-R R Suppl. 14"
```

=== Builder Strategy

The builder uses `build_supplement()` method to construct supplement identifiers:

[source,ruby]
----
def build_supplement(parsed_hash)
  # Extract supplement data
  supplement_data = parsed_hash[:supplement]
  
  # Build base identifier if present
  base = build(parsed_hash.reject { |k| k == :supplement })
  
  # Construct supplement with base
  supplement_klass.new(
    base: base,
    number: supplement_data[:number],
    date: cast(:date, supplement_data[:date])
  )
end
----

== Known Limitations

=== Combined Identifiers (6 failures)

**Pattern:** `ITU-T G.780/Y.1351 Amd 1 (2004)`

**Issue:** Parslet overwrites first identifier with second during parsing.

**Impact:** 6 tests fail (3 Amendment, 3 Corrigendum)

**Workaround:** Parse as individual identifiers:
```ruby
# Instead of combined:
id = PubidNew::Itu.parse("ITU-T G.780/Y.1351 Amd 1")  # Fails

# Parse separately:
id1 = PubidNew::Itu.parse("ITU-T G.780 Amd 1")  # Works
id2 = PubidNew::Itu.parse("ITU-T Y.1351 Amd 1")  # Works
```

**Future Enhancement:** Implement `CombinedIdentifier` wrapper class (like IEC's `VapIdentifier`) to handle these patterns properly.

**Status:** Acceptable for production use (edge case, <4% of tests)

== Design Principles

=== MODEL-DRIVEN Architecture

* **Identifiers are objects**, not strings
* Components encapsulate specific attributes
* Builder performs casting only, no business logic
* Identifiers handle rendering via `to_s`

=== MECE Organization

* Each identifier class handles mutually exclusive patterns
* Supplement base class for Amendment/Corrigendum/Supplement
* No pattern overlap between classes

=== Round-Trip Parsing

* Parse → Object → String must match original
* Components preserve original format
* 96.5% round-trip success rate

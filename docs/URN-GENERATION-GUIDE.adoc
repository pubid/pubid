= ISO URN Generation Guide
:toc:
:toc-title: Table of Contents

== Overview

PubID V2 implements RFC 5141-bis compliant URN generation for ISO identifiers.

RFC 5141-bis extends the original RFC 5141 (March 2008) with modern ISO standards support, including:

* Explicit language specification
* Dynamic copublisher combinations
* Extended document types
* Typed and harmonized stage codes
* Multi-level supplement support

== Quick Start

[source,ruby]
----
require 'pubid_new/iso'

# Parse and generate URN
id = PubidNew::Iso.parse("ISO 8601:2019")
id.to_urn  # => "urn:iso:std:iso:8601:ed-1:en"

# With copublisher
id = PubidNew::Iso.parse("ISO/IEC 27001:2013")
id.to_urn  # => "urn:iso:std:iso-iec:27001:ed-1:en"

# Technical Report
id = PubidNew::Iso.parse("ISO TR 9241-331:2012")
id.to_urn  # => "urn:iso:std:iso:tr:9241:-331:ed-1:en"

# Amendment
id = PubidNew::Iso.parse("ISO 8601:2019/Amd 1:2023")
id.to_urn  # => "urn:iso:std:iso:8601:ed-1:en:amd:2023:v1"
----

== Component Reference

=== Originator (Publisher)

The originator component identifies the publishing organization(s).

[cols="1,2,2"]
|===
|Pattern |Example |URN Component

|Single publisher
|ISO 8601:2019
|`iso`

|Copublisher
|ISO/IEC 27001:2013
|`iso-iec`

|Triple copublisher
|ISO/IEC/IEEE 29148:2018
|`iso-iec-ieee`

|Other copublishers
|ISO/ASTM 52901:2017
|`iso-astm`
|===

**Rules:**

* Always lowercase
* Hyphen-separated
* Preserves original order (not alphabetical)
* All ISO copublisher combinations supported (IEC, IEEE, ASTM, CIE, HL7, SAE, OECD, UNDP, etc.)

=== Type Component

Document type for non-International Standards.

[cols="1,2,2"]
|===
|Type |Abbreviation |URN Component

|Technical Report
|TR
|`tr`

|Technical Specification
|TS
|`ts`

|Guide
|Guide
|`guide`

|Directives
|DIR
|`dir`

|Directives Supplement
|DIR SUP
|`dir-sup`

|IWA Supplement
|IWA SUP
|`iwa-sup`

|Technology Trends Assessments
|TTA
|`tta`

|Publicly Available Specification
|PAS
|`pas`
|===

**Note:** International Standard (IS) is default and omitted from URN.

=== Number and Part

Document number with optional parts and subparts.

[cols="2,2"]
|===
|Identifier |URN Component

|ISO 8601:2019
|`8601`

|ISO 8601-1:2019
|`8601:-1`

|ISO 8601-1-2:2019
|`8601:-1-2`
|===

**Format:** Number followed by `:-X` for parts, `:-X-Y` for subparts.

Development stage for non-published documents.

Explicit abbreviations for common stages.

|===
|Stage |Abbreviation |URN Component

|WD
|`WD`

|Committee Draft
|CD
|`CD`

|Draft International Standard
|DIS
|`DIS`

|Final Draft International Standard
|FDIS
|`FDIS`

|Proposed Draft Amendment
|PDAM
|`PDAM`

|Draft Amendment
|DAM
|`DAM`

|Final Draft Amendment
|FDAM
|`FDAM`

|Draft Corrigendum
|DCOR
|`DCOR`

|Final Draft Corrigendum
|FDCOR
|`FDCOR`

|Committee Draft Technical Specification
|CDTS
|`CDTS`

|Draft Technical Specification
|DTS
|`DTS`

|Final Draft Technical Specification
|FDTS
|`FDTS`
|===

**Stage Iterations:** Append `.N` for iteration number (e.g., `FDIS.2`, `FDAM.3`)

==== Harmonized Stage Codes

Generic stage codes for unmapped stages using ISO Harmonized Stage Code system.

[cols="1,2,2"]
|===
|Stage |Harmonized Code |URN Component

|Preliminary
|00.00
|`stage-00.00`

|Proposal
|10.00
|`stage-10.00`

|Preparatory
|20.00
|`stage-20.00`

|Committee
|30.00
|`stage-30.00`

|Enquiry
|40.00
|`stage-40.00`

|Approval
|50.00
|`stage-50.00`
|===

**Stage Iterations:** For base identifiers, append `.vN` (e.g., `stage-30.00.v2`)

**Note:** Published documents (60.00, 60.60) omit stage component entirely.

=== Edition Component

Document edition number.

[cols="2,2"]
|===
|Identifier |URN Component

|ISO 8601:2019 ED1
|`ed-1`

|ISO 8601:2019 ED2
|`ed-2`
|===

**Format:** `ed-N` where N is the edition number.

=== Language Component

Explicit language specification per RFC 5141-bis principle: "explicit is better than implicit".

[cols="2,2"]
|===
|Identifier |URN Component

|ISO 8601:2019(en)
|`en`

|ISO 8601:2019(E)
|`en`

|ISO 8601:2019(en,fr)
|`en,fr`

|ISO 8601:2019(F)
|`fr`
|===

**Supported languages:**

* `en` - English
* `fr` - French  
* `ru` - Russian
* `ar` - Arabic
* `es` - Spanish
* `de` - German

**RFC 5141-bis Principle:** Always explicit, even for English documents.

=== Supplement Components

Amendments, corrigenda, and supplements.

[cols="1,2,2"]
|===
|Type |Abbreviation |URN Component

|Amendment
|Amd
|`amd`

|Corrigendum
|Cor
|`cor`

|Supplement
|Suppl
|`sup`

|Addendum (legacy)
|Add
|`sup`
|===

**Format with date:** `type:year:vN` (e.g., `amd:2023:v1`)

**Format without date:** `type:N:v1` (e.g., `amd:1:v1`)

**Version iterations:** Append `.N` for iteration (e.g., `v1.2`)

== RFC 5141-bis Extensions

=== Dynamic Copublishers

All ISO copublisher combinations supported:

* ISO/IEC (International Electrotechnical Commission)
* ISO/IEC/IEEE (Institute of Electrical and Electronics Engineers)
* ISO/ASTM (American Society for Testing and Materials)
* ISO/CIE (International Commission on Illumination)
* ISO/HL7 (Health Level 7)
* ISO/SAE (Society of Automotive Engineers)
* ISO/OECD (Organisation for Economic Co-operation and Development)
* ISO/UNDP (United Nations Development Programme)

**Format:** Lowercase, hyphen-separated, preserves original order.

**Example:**
[source,ruby]
----
PubidNew::Iso.parse("ISO/IEC/IEEE 29148:2018").to_urn
# => "urn:iso:std:iso-iec-ieee:29148:ed-1:en"
----

=== Extended Document Types

Beyond original RFC 5141 (2008):

* `dir` - Directives
* `dir-sup` - Directives Supplement  
* `iwa-sup` - IWA Supplement
* `tta` - Technology Trends Assessments

**Example:**
[source,ruby]
----
PubidNew::Iso.parse("ISO/IEC DIR 1:2022").to_urn
# => "urn:iso:doc:iso-iec:dir:1:2022"
----

=== Typed Stage Codes

Full range of ISO development stages:

**Base documents:**
* WD, CD, DIS, FDIS

**Amendments:**
* PDAM, DAM, FDAM

**Corrigenda:**
* DCOR, FDCOR

**Technical Specifications:**
* CDTS, DTS, FDTS

**Example:**
[source,ruby]
----
PubidNew::Iso.parse("ISO/FDAM 8601-1").to_urn
# => "urn:iso:std:iso:8601:-1:FDAM:amd:v1"
----

=== Harmonized Stage Codes

Generic `stage-XX.XX` format for unmapped stages using ISO Harmonized Stage Code system:

* `stage-00.00` - PWI (Preliminary Work Item)
* `stage-10.00` - NP, NWIP (New Work Item Proposal)
* `stage-20.00` - AWI, WD (Approved Work Item)
* `stage-30.00` - CD (Committee Draft)
* `stage-40.00` - DIS (Draft International Standard)
* `stage-50.00` - FDIS (Final Draft International Standard)

**Published stages filtered:** 60.00, 60.60 (omitted from URN as they represent published state).

**Example:**
[source,ruby]
----
PubidNew::Iso.parse("ISO/PWI 12345").to_urn
# => "urn:iso:std:iso:12345:stage-00.00"

PubidNew::Iso.parse("ISO/NP 12345").to_urn
# => "urn:iso:std:iso:12345:stage-10.00"
----

=== Explicit Language Specification

Always includes language codes, following principle: **"Explicit is better than implicit"**

Even English documents include `:en` for clarity and consistency.

**Example:**
[source,ruby]
----
PubidNew::Iso.parse("ISO 8601:2019(en)").to_urn
# => "urn:iso:std:iso:8601:ed-1:en"

PubidNew::Iso.parse("ISO 8601:2019(fr)").to_urn
# => "urn:iso:std:iso:8601:ed-1:fr"

PubidNew::Iso.parse("ISO 8601:2019(en,fr)").to_urn
# => "urn:iso:std:iso:8601:ed-1:en,fr"
----

=== Multi-Level Supplement Support

Properly handles nested supplement chains:

* Base document context preserved
* All supplements flattened in order
* Each supplement includes type, year, version

**Example:**
[source,ruby]
----
PubidNew::Iso.parse("ISO/IEC 13818-1:2015/Amd 3:2016/Cor 1:2017").to_urn
# => "urn:iso:std:iso-iec:13818:-1:amd:2016:v3:cor:2017:v1"
----

**Components preserved:**

* Base: `iso-iec:13818:-1`
* Amendment 3 (2016): `amd:2016:v3`
* Corrigendum 1 (2017): `cor:2017:v1`

== Known Limitations

=== Acceptable Differences from V1

These differences represent improvements in V2:

==== 1. Explicit Language Codes

**V1:** May omit language code
**V2:** Always explicit (more correct)

[source,ruby]
----
PubidNew::Iso.parse("ISO 8601:2019(en)").to_urn
# V1: "urn:iso:std:iso:8601:ed-1"
# V2: "urn:iso:std:iso:8601:ed-1:en"
----

**Rationale:** RFC 5141-bis principle of "explicit over implicit".

==== 2. Specific Harmonized Codes

**V1:** Generic "stage-draft"
**V2:** Specific "stage-40.00" (more informative)

[source,ruby]
----
PubidNew::Iso.parse("ISO/DIS 12345").to_urn
# V1 generic: "urn:iso:std:iso:12345:stage-draft"
# V2 specific: "urn:iso:std:iso:12345:DIS"
----

**Rationale:** More informative, follows ISO Harmonized Stage Codes.

==== 3. Base Identifier Context

**V1:** May lose context in multi-level supplements
**V2:** Preserves full context (more accurate)

[source,ruby]
----
PubidNew::Iso.parse("ISO/IEC 13818-1:2015/Amd 3:2016/Cor 1:2017").to_urn
# V2: "urn:iso:std:iso-iec:13818:-1:amd:2016:v3:cor:2017:v1"
#     ^^^^^^^^^^^^^^^^^^^^^^^^^ Full base context preserved
----

**Rationale:** Unambiguous identification of exact document.

=== RFC 5141-bis Ambiguities

Some patterns have ambiguous specification:

==== 1. PRF (Proof) Stage

**Issue:** Stage 60.00 is technically published, but some expect stage component.

**V2 approach:** Filters as published (conservative).

[source,ruby]
----
PubidNew::Iso.parse("ISO/PRF 6709:2022").to_urn
# => "urn:iso:std:iso:6709"
----

**Rationale:** Stage 60.00 is published. Omitting stage is more correct per RFC 5141 semantics.

==== 2. Bundled Identifier Format

**Issue:** Limited RFC 5141 guidance on bundled identifiers.

**V2 approach:** Flattened component approach.

[source,ruby]
----
PubidNew::Iso.parse("ISO/IEC DIR 1:2022 + IEC SUP:2022").to_urn
# => "urn:iso:doc:iso-iec:dir:1:2022:iec:sup:2022"
----

**Rationale:** Maintains readability while preserving all components.

==== 3. Legacy Format Conversions

**Issue:** V2 normalizes to modern format.

**V2 approach:** Consistent modern format.

[source,ruby]
----
PubidNew::Iso.parse("ISO 8601:2019 Ed 1").to_urn
# V1: May preserve "Ed 1"
# V2: Normalizes to "ed-1"
----

**Rationale:** Consistent, predictable format.

== Troubleshooting

=== Missing Stage in URN

**Problem:** Published document has no stage component in URN.

**Explanation:** Documents at stage 60.00 or 60.60 are published. RFC 5141-bis omits stage for published documents as they are the default state.

**Solution:** This is correct behavior. Published documents don't need stage specification.

=== Wrong Type Code

**Problem:** Type code doesn't match expectation.

**Solution:** Check TYPED_STAGES in identifier class. Type code comes from `TypedStage.type_code` attribute. Use `identifier.typed_stage.type_code` to verify.

=== Multi-Level Supplement Issues

**Problem:** Base context lost in nested supplements.

**Solution:** Ensure proper SupplementIdentifier chain. Each supplement must have `base_identifier` attribute pointing to previous level. V2 walks the entire chain to preserve context.

=== Language Code Missing

**Problem:** Language code not in URN.

**Solution:** Check `identifier.languages` attribute. Language must be parsed from original identifier string: `ISO 8601:2019(en)`.

=== Stage Iteration Placement

**Problem:** Stage iteration in wrong place.

**Explanation:**

* **Typed stages (base):** Iteration in stage code (`FDAM.2`)
* **Harmonized codes (base):** Iteration in stage code (`stage-30.00.v2`)
* **Harmonized codes (supplement):** Iteration in version part (`v1.2`)

**Solution:** Check if identifier is a supplement. Different rules apply.

== Implementation Notes

=== Architecture

**Component:** `lib/pubid_new/iso/urn_generator.rb`

**Entry point:** `identifier.to_urn` method

**Pattern:** Separate generator class (not in identifier itself)

**Benefits:**

* Clear separation of concerns
* Easier to test
* Can be extended independently

=== Stage Handling

**Typed stages:** `TYPED_STAGE_MAP` for explicit abbreviations

**Harmonized stages:** `harmonized_stages` attribute fallback

**Published:** Filtered (60.00, 60.60)

**Code:**
[source,ruby]
----
# Try typed stage abbreviations first
if TYPED_STAGE_MAP.key?(stage_code)
  return TYPED_STAGE_MAP[stage_code]
end

# Fallback to harmonized codes
harmonized_codes = identifier.typed_stage.harmonized_stages
harmonized_code = harmonized_codes.first
"stage-#{harmonized_code}"
----

=== Iteration Placement

**Typed stages (base):** In stage code (`FDAM.2`)

**Harmonized codes (base):** In stage code (`stage-30.00.v2`)

**Harmonized codes (supplement):** In version part (`v1.2`)

**Code:**
[source,ruby]
----
# For base identifiers with harmonized codes
if identifier.stage_iteration && !identifier.is_a?(SupplementIdentifier)
  stage_part += ".v#{identifier.stage_iteration.value}"
end
----

=== Design Decisions

==== 1. RFC 5141-bis Only

**Decision:** No dual-mode complexity.

**Rationale:** Single-focus implementation is cleaner, faster, easier to maintain.

==== 2. Explicit Over Implicit

**Decision:** Language codes always included.

**Rationale:** RFC 5141-bis guidance: "explicit is better than implicit".

==== 3. Specific Over Generic

**Decision:** Use specific harmonized codes (stage-40.00) not generic (stage-draft).

**Rationale:** More informative, follows ISO Harmonized Stage Codes.

==== 4. Component-Based

**Decision:** Each URN part from proper component.

**Rationale:** No hardcoded strings, maintainable architecture.

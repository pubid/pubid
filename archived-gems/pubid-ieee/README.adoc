= IEEE publication identifiers ("IEEE PubID")

== Purpose

Implements a mechanism to parse and utilize IEEE publication identifers.

== Historic identifier patterns

There are at least two major "pattern series" of identifiers due to historical reasons: old (type I) and new (type II).
This implementation attempts to support both types of publication identifier patterns.

== Use cases to support

* analyze a pattern of type I idetifier
* parse type II idetifier into components
* generate a filename from the components similar to type I pattern

== Elements of the PubID

=== Publisher

|===
| Name | Abbrev

| Institute of Electrical and Electronics Engineers
| IEEE

|===

=== Report number

`{number}` - is a set of one or more digits and optional letters

=== Part

`{part}` - is a set of digits and optional letters; starts with a digit; if a letter or letters are present then they are in the end; optional

=== Subpart

`{subpart}` - is a set of digits and optional letters; optional, many subparts are possible

=== Year

`{year}` - is a set of 4 digits; optional

=== Corrigendum & Amendment

`{cor}` - is a corrigendum or an amendments with the pattern `Cor {cornum}-{year}` or `Amd {cornum}:{year}` where {cornum} is a set of digits; optional

== Type I pattern

[source]
----
{publisher} {type} {series} {number}{part}.{subpart}{year} {edition}/{conform}/{correction}
----

* `{publisher}` IEEE
* `{type}` one of the values: `Standard`, `Std`, `Draft`, `Draft Standard`, `Draft Supplement` ^*^
* `{series}` one of the values: `ISO/IEC`, `ISO/IEC/IEEE` ^*^
* `{number}` set of digits optionally prefixed with uppercase letter and optionally suffixed  with letter
* `{part}` from 1 to 2 digits prefixed with `.` or `-` and optionally suffixed with up to 4 letters ^*^
* `{subpart}` 1 digit optionally suffixed with a letter ^*^
* `{year}` 4 digits prefixed with `-`, `:`, ` - `, or breakspace ^*^
* `{edition}` prefix `Edition` followed by a reference in brackets or prefix `First edition` followed by date in format `YYYY-MM-DD` ^*^
* `{conform}` prefix `Conformance` followed by 2 digits, dash, and 4 digits year ^*^
* `{correction}` prefix `Cor` optionally followed by breakspace, or prefix `Amd` followed by `.`, followed by from 1 to 2 digits, dash and 4 digits year ^*^

(*) - optional

An identifier can be composed of 2 other identifiers with breakspace delimiter. Only the first identifier needs to cnatain puplisher, for the secont it's optional

Following RegEx expression parses 100% of identifiers from the type I https://xml2rfc.tools.ietf.org/public/rfc/bibxml-ieee/[dataset]:
[source,regex]

----
{
  ^IEEE\s
  ((?<type1>Standard|Std|Draft(\sStandard|\sSupplement)?)\s)?
  ((?<series>ISO\/IEC(\/IEEE)?)\s)?
  (?<number1>[A-Z]?\d+[[:alpha:]]?)
  ([.-](?<part1>\d{1,2}(?!\d)[[:alpha:]]{0,4}))?
  (\.(?<subpart1>\d[[:alpha:]]?))?
  (?<year1>([-:]|\s-\s|,\s)\d{4})?
  (\s(IEEE\s(?<type2>Std)\s)?(?<number2>[A-Z]?\d+[[:alpha:]]?)
    ([.-](?<part2>\d{1,2}(?!\d)[[:alpha:]]{0,4}))?
    ([.](?<subpart2>\d[[:alpha:]]?))?
    (?<year2>([-:.]|_-|\s-\s|,\s)\d{4})?)?
  (\s(?<edition>Edition(\s\([^)]+\))?|First\sedition\s[\d-]+))?
  (\/(?<conform>Conformance\d{2})-(?<confyear>\d{4}))?
  (\/(?<correction>(Cor\s?|(Amd\.)\d{1,2})
    (?<coryear>(:|-|:-)\d{4}))?$
}x
----

== Pasing PubID elements from type II identifiers

To parse PubID elements from the type II pattern identifiers we can use a RegEx expression:

[source,regex]
----
{
  ^IEEE\s(?<number1>\w+(\.[A-Z]\d|\sHBK)?)
  (?<part1>(\.|\s)\d{1,4}[[:alpha:],]{0,7}|-\d?[A-Z]+|-\d(?=[-.]))?
  (?<subpart11>\.\d{1,3}[a-z]?|-\d{5}[a-z]?|-\d+(?=[-:_]))?
  (?<subpart12>\.\d|-\d+(?=-))?
  (?<year1>([-:.]|_-|\s-)\d{4})?
  (\/(?<number2>([A-Z]?\d+[a-z]?|Conformance\d+))
    ((\.|-)(?<part2>\d{1,3}[a-z]?)(?!\d))?
    (\.(?<subpart21>\d{1,2}))?)?
  (\/(?<number3>\d+)(\.(?<part3>\d))?)?
  (?<year2>([-:.]|_-|\s-)\d{4})?
  ((\/|_|-|\s\/)(?<correction>(Cor|(?i)Amd(?-i))(\s|\.|\.\s)?\d{1,2})
    (?<coryear>(:|-|:-|_[A-Z][a-z]{2}_)\d{4}(-\d{4})?)?)?$
}x
----

This RegEx expession covers 99% of the identifiers from the type II bibxml-ieee https://xml2rfc.tools.ietf.org/public/rfc/bibxml-ieee-new/[dataset].

== File name generator

For type I identifiers file names are generated by replacing symbols `/`, `\`, `,`, `'`, `"`, `(`, `)`, and breakspace with symbol `_`. Sequences of multiple sybols `_` should be squized to one symbol.

For type II identifiers it needs to parse PubID elements than join the elements in order:

----
IEEE.{number1}_{part1}.{subpart11}.{subpart12}-{year1}_{number2}_{part2}.{subpart21}_{number3}_{part3}-{year2}_{correction}-{coryear}
----

== V2 implementation

=== General

The IEEE V2 parser has been completely redesigned with a clean three-layer architecture and model-driven design, achieving **100% success rate** on complex test patterns.

The V2 implementation is located in [`lib/pubid/ieee/`](../../lib/pubid/ieee/) while V1 code remains for backward compatibility during the migration period.

=== Architecture

==== Three-layer design

V2 implements clean separation of concerns across three distinct layers:

[source]
----
┌─────────────────────────────────────┐
│         Parser Layer                │  Parslet grammar (syntax rules)
│  (Syntax → Parse Tree)              │
└──────────┬──────────────────────────┘
           │
┌──────────▼──────────────────────────┐
│         Builder Layer               │  Parse tree transformation
│  (Parse Tree → Attributes)          │
└──────────┬──────────────────────────┘
           │
┌──────────▼──────────────────────────┐
│      Identifier Layer               │  Lutaml::Model classes
│  (Attributes → String)              │
└─────────────────────────────────────┘
----

Where:

Parser:: [`parser.rb`](../../lib/pubid/ieee/parser.rb) - Parslet-based grammar defining syntax rules (~210 lines)
Builder:: [`builder.rb`](../../lib/pubid/ieee/builder.rb) - Transforms parse tree to attributes (~330 lines)
Identifier:: [`identifiers/`](../../lib/pubid/ieee/identifiers/) - Lutaml::Model classes with rendering logic
Components:: [`components/`](../../lib/pubid/ieee/components/) - Reusable components (Code, Draft)

==== Identifier classes

The V2 implementation uses dedicated identifier classes for different publication patterns:

Base:: [`identifiers/base.rb`](../../lib/pubid/ieee/identifiers/base.rb) - Standard IEEE Std, AIEE, ANSI formats
AdoptedStandard:: [`identifiers/adopted_standard.rb`](../../lib/pubid/ieee/identifiers/adopted_standard.rb) - Standards adopted in parenthetical notation
DualPublished:: [`identifiers/dual_published.rb`](../../lib/pubid/ieee/identifiers/dual_published.rb) - Space-separated and " and " patterns
IecIeeeCopublished:: Standards co-published between IEC and IEEE

Each class handles its specific notation patterns and rendering requirements.

==== Components

Code:: [`components/code.rb`](../../lib/pubid/ieee/components/code.rb) - Handles code parsing with automatic separator preservation
Draft:: [`components/draft.rb`](../../lib/pubid/ieee/components/draft.rb) - Draft version, revision, and date handling

=== V2 features

==== IEC as standalone publisher

IEC identifiers are now fully supported as standalone publishers.

.Parsing IEC identifiers
[example]
====
[source,ruby]
----
# Basic IEC identifier
id = Pubid::Ieee.parse("IEC 61523-4")
id.to_s  # => "IEC 61523-4"

# IEC with edition and year-month
id = Pubid::Ieee.parse("IEC 61671-2 Edition 1.0 2016-04")
id.to_s  # => "IEC 61671-2 Edition 1.0 2016-04"
----
====

**Implementation:** Added IEC, AESC to organizations in [`parser.rb:40`](../../lib/pubid/ieee/parser.rb#L40)

==== Parenthetical content preservation

Parenthetical notes and references are now preserved exactly as parsed.

.Parsing identifiers with parenthetical content
[example]
====
[source,ruby]
----
# Supersession note
id = Pubid::Ieee.parse("AIEE No 19-1943 (Supercedes A. I. E. E. Standard No. 19-1938)")
id.to_s  # => "AIEE No 19-1943 (Supercedes A. I. E. E. Standard No. 19-1938)"

# Revision note
id = Pubid::Ieee.parse("IEEE Std 1018-2013 (Revision of IEEE Std 1018-2004)")
id.to_s  # => "IEEE Std 1018-2013 (Revision of IEEE Std 1018-2004)"

# Amendment note
id = Pubid::Ieee.parse("IEEE Std 802.16m-2011 (Amendment to IEEE Std 802.16-2009)")
id.to_s  # => "IEEE Std 802.16m-2011 (Amendment to IEEE Std 802.16-2009)"
----
====

**Implementation:**
- Enhanced additional_parameters rule in [`parser.rb:160`](../../lib/pubid/ieee/parser.rb#L160)
- Extract parenthetical_content in [`builder.rb:318`](../../lib/pubid/ieee/builder.rb#L318)
- Added parenthetical_content attribute in [`identifiers/base.rb:34`](../../lib/pubid/ieee/identifiers/base.rb#L34)
- Render in to_s() in [`identifiers/base.rb:165`](../../lib/pubid/ieee/identifiers/base.rb#L165)

==== Multi-part adoptions

IEEE standards adopting multiple external standards are supported.

.Parsing multi-part adoption identifiers
[example]
====
[source,ruby]
----

id = Pubid::Ieee.parse("IEEE Std 623-1976 (ANSI Y32.21-1976, NCTA 006-0975)")

# AIEE adopting AESC standard
id = Pubid::Ieee.parse("AIEE No 14-1925 (AESC C22-1925)")
id.to_s  # => "AIEE No 14-1925 (AESC C22-1925)"
----
====

**Implementation:** Skip recursive parsing when commas present in [`identifiers/base.rb:130`](../../lib/pubid/ieee/identifiers/base.rb#L130)

==== Dash preservation in codes

The Code component now preserves the original separator (dash or dot) used in the input.

.Parsing codes with dash preservation
[example]
====
[source,ruby]
----
# Dash-separated code
id = Pubid::Ieee.parse("AESC C22-1925")
id.code.original_separator  # => "-"
id.code.to_s                # => "C22-1925"  # dash preserved

# Dot-separated code
id = Pubid::Ieee.parse("IEEE Std C37.111-2013")
id.code.original_separator  # => "."
id.code.to_s                # => "C37.111"  # dot preserved
----
====

**Implementation:**
- Added original_separator attribute to Code component in [`components/code.rb:18`](../../lib/pubid/ieee/components/code.rb#L18)
- Parser detects and stores actual separator (dash or dot)
- Rendering uses preserved separator in [`components/code.rb:48`](../../lib/pubid/ieee/components/code.rb#L48)

==== Space-separated dual identifiers

Standards jointly published by two organizations with space-separated format are automatically detected and normalized.

.Parsing space-separated dual identifiers
[example]
====
[source,ruby]
----
# Space-separated input (auto-detected)
id = Pubid::Ieee.parse("IEC 62014-5 IEEE Std 1734-2011")
id.class  # => Pubid::Ieee::Identifiers::DualPublished
id.to_s   # => "IEC 62014-5 and IEEE Std 1734-2011"  # normalized with "and"

# " and " format (preserved)
id = Pubid::Ieee.parse("ANSI C37.61-1973 and IEEE Std 321-1973")
id.to_s   # => "ANSI C37.61-1973 and IEEE Std 321-1973"
----
====

**Implementation:**
- Dynamic publisher position detection in [`identifiers/base.rb:113`](../../lib/pubid/ieee/identifiers/base.rb#L113)
- Distinguishes from co-published (Publisher1/Publisher2)
- Comprehensive tests in [`spec/pubid/ieee/identifiers/dual_published_spec.rb`](../../spec/pubid/ieee/identifiers/dual_published_spec.rb)

==== Code separator logic

Smart separator selection based on code patterns (dots vs dashes).

.Code pattern examples
[example]
====
[source]
----
C37.111        # Letter prefix → dots
61523-4        # Long number (5+ digits) → dashes
P11073-10404   # P-prefix → dashes
802.11         # Traditional IEEE → dots
----
====

**Implementation:** Smart separator logic in [`components/code.rb:48`](../../lib/pubid/ieee/components/code.rb#L48)

=== Edge cases solved

The V2 implementation successfully handles these previously problematic patterns:

==== Dash preservation

Codes like `C22-1925` now correctly preserve dashes instead of converting to dots.

**Before:** `AIEE No 14-1925 (AESC C22-1925)` → `AIEE No 14-1925 (AESC C22.1925)` ❌

**After:** `AIEE No 14-1925 (AESC C22-1925)` → `AIEE No 14-1925 (AESC C22-1925)` ✅

==== Space-separated dual identifiers

Space-separated patterns are automatically detected and normalized.

**Input:** `IEC 62014-5 IEEE Std 1734-2011`

**Output:** `IEC 62014-5 and IEEE Std 1734-2011` ✅

==== Multi-publisher rendering

Proper slash and comma rendering for co-published standards.

**Before:** Extra slashes and commas in output ❌

**After:** Clean rendering: `ISO/IEC/IEEE P90003` ✅
